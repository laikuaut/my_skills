# 論理設計フェーズ オーケストレータ

新規プロダクトの**要求整理 → 設計 → 実装着手前**までを一気通貫で進める手順管理スキル。
このスキル自身は設計書の細部書式を持たず、テンプレートは `references/` に分離し、実装・テスト・雛形生成は他スキルに委譲する。

`fullstack-pipeline` が "実装フェーズ" を指揮するのに対し、本スキルは **"論理設計フェーズ"** を指揮する。直列に使うと要求から実装完成まで一気通貫になる。

## 責務分担

| このスキル | 委譲先 |
|-----------|--------|
| 要求整理のファシリテーション | — |
| 設計計画の立案・進捗管理 | — |
| 設計ドキュメント13本の構成・依存関係 | — |
| 各フェーズの遷移とユーザ承認 | — |
| 実装コード・型定義の品質基準 | python-impl / react-impl |
| プロジェクト雛形の生成 | python-project-init / react-project-init |
| OpenAPIの自動生成 | openapi-docs（本スキルは手書きドラフトまで） |
| 画面モック（ビジュアル） | frontend-design プラグイン（別セッション推奨） |
| 画面仕様書の最終版（実装後リバース） | screen-spec-reverse |
| API結合試験 / E2Eテスト | rest-api-test / playwright-test |
| コミット・プッシュ | git-commit-push |

## 成果物構造

```
<project-root>/
├── README.md                        # プロジェクトの玄関
├── CLAUDE.md                        # Claude Code 向け指示書
└── docs/
    ├── requirements.md              # 要求仕様 v1.0
    ├── design-plan.md               # 設計タスク計画＋進捗
    ├── design/
    │   ├── README.md                # 設計書総目次（Mermaid依存図付き）
    │   ├── concept.md               # D1.1 UI/UXコンセプト
    │   ├── tokens.md                # D1.2 デザイントークン
    │   ├── components.md            # D1.3 コンポーネントカタログ
    │   ├── interactions.md          # D1.5 マイクロインタラクション
    │   ├── architecture.md          # D2.1 システム全体像
    │   ├── data-layer.md            # D2.2 データ層詳細
    │   ├── state-management.md      # D2.3 フロント状態管理
    │   ├── error-handling.md        # D2.4 エラーハンドリング
    │   ├── api-detail.md            # D3.1 API詳細
    │   ├── schemas.md               # D3.2 Pydantic/Zodスキーマ
    │   ├── mcp-spec.md              # D4.1 MCP仕様（任意）
    │   └── mcp-integration.md       # D4.2 MCP統合（任意）
    ├── openapi.yaml                 # D3.3 OpenAPI 3.1 ドラフト
    └── screens/                     # D5 画面仕様書（暫定 or リバース）
```

**D1.4 画面モック** は `frontend-design` プラグイン側で作るため、本スキル内では別セッションを案内するに留める。
**D4.x MCP系** は MCP 連携プロダクトのみ作成する。

## 発火時の最初のメッセージ

スキルが発火したらまず以下を提示する。

```
設計フェーズを開始します。

まず要求を整理していきます。いくつか質問させてください。

【Step 0: プロジェクトの輪郭】
- どんなプロダクトを作りますか？（1〜2行で）
- 想定ユーザーは誰ですか？
- 最優先する価値は？（UI/UX品質 / 機能数 / 性能 / プライバシー など）

回答をもとに、要求仕様（requirements.md）→ 設計計画（design-plan.md）→
設計書本編の順に作成していきます。途中で確認を取りながら進めます。
```

## ワークフロー

### Phase 0: プロジェクト定義

既存の `README.md` / `CLAUDE.md` が別テンプレの場合は、本プロダクト向けに書き換える。

- プロダクトの一行サマリ
- 技術スタック（フロント / バック / DB / MCP の有無）
- 現ステータス（"設計中" / "要求整理中" 等）

完了したら Phase 1 に進む旨をユーザに伝える。

### Phase 1: 要求仕様 → `docs/requirements.md`

**AskUserQuestion で段階的に確定する**。一気に全部聞かず、**2〜3問ずつ・選択肢は4つ以内・推奨案を先頭**に置くのが鉄則。選択疲れで離脱を招かないため。

質問セットは `references/ask-user-questions.md` を参照。典型的な確認項目:

- プロダクト目的・想定ユーザー
- 技術スタック（フロント / バック / DB / MCP有無）
- 対応言語（i18n）
- 認証の有無
- データ保存方式（DB / ファイル）
- エクスポート・インポートの要否
- デプロイ方針
- 想定ユーザー数
- インポート衝突解決（上書き / 新ID採番）
- タグ色プリセット有無
- 初期データ（空 / サンプル同梱）
- バックエンド単独起動の可否

requirements.md の章立ては `references/requirements-template.md` を使う。**確定事項は §10、未決事項は §11** に分離し、設計フェーズ中に順次解消する。

### Phase 2: 設計計画 → `docs/design-plan.md`

D1〜D5 の大項目リストを作成する。

- **D1**: UI/UXデザイン（D1.1 コンセプト / D1.2 トークン / D1.3 コンポーネント / D1.4 画面モック / D1.5 インタラクション）
- **D2**: システム設計（D2.1 アーキテクチャ / D2.2 データ層 / D2.3 状態管理 / D2.4 エラー）
- **D3**: API設計（D3.1 API詳細 / D3.2 スキーマ / D3.3 OpenAPI）
- **D4**: MCP設計（D4.1 仕様 / D4.2 統合）※MCP連携時のみ
- **D5**: 画面仕様書

テンプレは `references/design-plan-template.md`。以下を必ず含める:

- **進捗サマリ表**を冒頭に配置（各タスク・状態・備考）
- **スケジュール目安**（1項目=0.5〜2日の仮置き）
- **完了判定（DoD）** を定義
- **優先順位**（必須 / 後回し可）を明示

### Phase 3: 設計書総目次 → `docs/design/README.md`

以下を含める（テンプレは `references/design-docs-templates/README.md`）:

- 全13ドキュメントの一覧表（状態列付き）
- **Mermaid依存関係図**（requirements → concept → tokens → … → openapi.yaml）
- 役割別の読み順（**初見 / フロント / バック / MCP / レビュアー** の5パターン）
- **§重要な横断決定事項**（空で始め、各フェーズで追記していく）
- メンテナンスルール（追加・更新の責務）

Mermaid記法の落とし穴は `references/mermaid-tips.md`。特にシーケンス図では `<br/>` + `"..."` + `;` + `{}` の組み合わせはパーサーエラーになりやすく、特殊文字は `Note over` に逃がす。

### Phase 4: 設計書本編

各ドキュメントの**ひな形は `references/design-docs-templates/` にある**。共通構造:

1. 冒頭: 対象・作成日・フェーズID・関連ドキュメントへのリンク
2. 設計原則（箇条書き）
3. 詳細仕様（**型定義・関数シグネチャ・エラー時挙動まで書く**）
4. テスト戦略
5. 実装順序
6. 次のドキュメントへの引継ぎメモ

**粒度の基準**: 「このまま実装に落とせるか？」を自問する。曖昧な文章は避け、数字・型・挙動を確定させる（例: 「1行36px・2px行間・1画面25件以上」「staleTime=0 / refetchOnWindowFocus=true」「サンプルデータ: タグ4件 + タスク8件」）。

#### タスク推奨順序

```
D1.1 コンセプト ──┬── D1.2 トークン ── D1.3 コンポーネント ──┬── D1.5 インタラクション
                 │                                          └── D1.4 画面モック（別セッション推奨）
                 │
D2.1 アーキ ─────┼── D2.2 データ層
                 ├── D2.3 状態管理
                 ├── D2.4 エラー
                 └── D3.1 API詳細 ── D3.2 スキーマ ── D3.3 OpenAPI
                     └── D4.1 MCP仕様 ── D4.2 MCP統合
```

- D1.1 と D2.1 は**並行可**（要求からの最初の分岐）
- **D1.4 画面モックは別セッションで `frontend-design` プラグインを使う**
- D4.x は MCP連携プロダクトの場合のみ作成

### Phase 5: 各タスク完了時の更新プロトコル（省略禁止）

ドキュメントが1つ完成するたびに以下を**必ず**実行する。ここを飛ばすと進捗が崩壊する。

1. 成果物ファイルを作成 / 更新
2. `docs/design-plan.md` の進捗サマリ表で該当行を ✅ に
3. 「Dx.y で確定した重要事項」を design-plan.md に追記（後続タスクへの引継ぎ）
4. `docs/design/README.md` の一覧表で該当行を ✅ に、進捗数字を更新
5. **横断的な決定**（UI原則・アーキ原則・データ層原則など）は `design/README.md §重要な横断決定事項` に追記
6. 要求仕様が変わった場合は **requirements.md を先に**更新してから設計に反映

個別ドキュメントに書く詳細と、README に書く横断要約を分離することで、全体像と詳細の両方を保てる。

### Phase 6: セッション分割の判定

長時間セッションを避けるため、区切りごとに継続判定する。詳細は `references/session-splitting.md`。

| タイミング | 判定 |
|---|---|
| 要求仕様確定後 | 区切り候補 |
| D1.1 + D2.1（初期設計）完了後 | 区切り候補 |
| システム設計（D2.x）完了後 | 区切り候補 |
| API設計（D3.x）完了後 | **強い区切り推奨** |
| D1.2〜D1.5（デザイン系）着手前 | **新セッション推奨**（`frontend-design`） |

**基準**:
- コンテキスト使用量が 40% を超えたら切替を検討
- デザイン系（視覚設計）とシステム設計は**別セッション**が効率的
- D1.4 画面モックは**専用セッション**

### Phase 7: コミット

区切りごとに `git-commit-push` スキルを呼び出す。推奨分割:

- 成果物ドキュメント = 機能領域単位でコミット（要求／設計計画／UI系／システム系／API系）
- 計画・目次の更新 = 最後に1コミットで進捗反映

## 重要パターン

`references/anti-patterns.md` に詳述。要点のみ:

1. **AskUserQuestion は段階確定** — 2〜3問ずつ・選択肢4つ以内・推奨案先頭
2. **確定事項と未決事項を分離** — 確定は表、未決はチェックリスト
3. **引継ぎメモを蓄積** — design-plan.md に「Dx.y で確定した重要事項」を追加
4. **横断決定を一元管理** — design/README.md §重要な横断決定事項にカテゴリ別集約、個別ドキュメントは具体仕様のみ
5. **Mermaid図は要所で** — 目次依存関係・データフロー・読み書きフロー
6. **粒度は「実装に落とせる」基準** — 型・関数シグネチャ・エラー挙動まで
7. **具体値を残す** — 曖昧な数字を避ける

## アンチパターン

| やらない | 理由 |
|---|---|
| 要求仕様と設計を同じドキュメントに混ぜる | 関心の分離が崩れる |
| 全13ドキュメントを1セッションで書き切ろうとする | コンテキスト肥大・集中力低下 |
| タスク完了時に design-plan.md を更新し忘れる | 進捗管理が崩壊 |
| 横断決定を個別ドキュメントだけに書く | 全体像が見えない |
| 実装寄りの詳細を書きすぎる | 設計書が肥大・保守困難 |
| ユーザー承認なしに要求を追加する | スコープクリープ |
| Mermaid図で特殊文字を無警戒に使う | レンダリングエラー |
| "サーバー状態" を Zustand に入れる設計 | キャッシュ二重管理で不整合 |
| 一度の AskUserQuestion で 5問以上聞く | 選択疲れ・離脱 |

## 既存スキルとの関係

| 既存スキル | 本スキルとの関係 |
|---|---|
| `fullstack-pipeline` | **後工程**（本スキル完了後に起動、実装を指揮） |
| `python-impl` / `react-impl` | 後工程（実装時に呼ばれる） |
| `python-project-init` / `react-project-init` | 本スキルは設計のみ。雛形生成は後工程 |
| `openapi-docs` | 本スキルは手書きドラフト、実装後に自動生成版と突合 |
| `rest-api-test` | 後工程（実装後のテスト生成） |
| `playwright-test` | 後工程（実装後のE2E） |
| `screen-spec-reverse` | 後工程（実装後に画面仕様書をリバース生成） |
| `git-commit-push` | 本スキル内で**区切りごとに呼ぶ** |
| `frontend-design` プラグイン | D1.4 画面モック作成時に**別セッションで**起動 |

**重要**: 本スキルは他スキルの領域に踏み込まない。雛形生成・実装・テストは各専用スキルに委譲する。

## 完了報告フォーマット

設計フェーズの区切りが来たら以下を表示する:

```markdown
## 設計フェーズ完了

作成したドキュメント: {N} / 13 本
所要ターン: {M} 回
主要な横断決定:
  - {決定1}
  - {決定2}
  ...

次のステップ:
  - 実装に進むなら: `fullstack-pipeline` スキル
  - 画面モックを作るなら: `frontend-design` プラグインを別セッションで
  - まだ未決のタスク: {D1.4 / D4.x / D5 など}
```

## スキル運用上の注意

1. **オーケストレーション専用** — 細部書式は `references/` に分離し、SKILL.md 本体は流れの管理に特化する
2. **ユーザー承認を必ず挟む** — 要求確定・設計方針変更・コミット・セッション切替などの判断タイミング
3. **既存スキルを呼ぶときは委譲** — 自分で実装せず、該当スキルが発火するよう案内する
4. **テンプレは強制しない** — プロジェクト性質によって不要なセクションもある。採用/不採用をユーザーに確認
