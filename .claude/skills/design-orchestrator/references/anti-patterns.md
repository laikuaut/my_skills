# アンチパターン集と回避策

設計フェーズで陥りがちな失敗と、なぜそれがダメか、どうすべきか。

## 1. 要求仕様と設計を同じドキュメントに混ぜる

**症状**: requirements.md に「この画面は React で」「API は /api/tasks」まで書かれている。

**なぜダメか**: 要求（何を・なぜ）と設計（どう）は読むタイミングも更新頻度も違う。混ぜると、要求が変わったときに設計が道連れにされ、逆も起こる。

**回避**:
- requirements.md は「何を、誰のために、なぜ」
- design/*.md は「どう実現するか」
- 分離すると **要求が変わったら requirements を先に更新 → 設計に波及** の流れが守れる

## 2. 全13ドキュメントを1セッションで書き切ろうとする

**症状**: 12時間セッションでコンテキストが肥大化し、最後の方は粒度が落ちる。

**なぜダメか**: LLMの集中力はコンテキストサイズに反比例する。特に**視覚設計とシステム設計は頭の使い方が違う**ので、同じセッションで切り替えると質が落ちる。

**回避**:
- `session-splitting.md` の区切り候補に従い、API設計完了 / システム設計完了で分割
- デザイン系（D1.2〜D1.5）は別セッション推奨
- D1.4 画面モックは **frontend-design プラグイン専用セッション**

## 3. タスク完了時に design-plan.md を更新し忘れる

**症状**: ドキュメントだけが増え、進捗サマリ表は ⬜ のまま。次セッションで「何が終わってるんだっけ」となる。

**なぜダメか**: 進捗管理は**整合性が崩れた瞬間に信頼性ゼロ**。一度崩れると「どれが最新？」の確認コストが膨大になる。

**回避**:
- SKILL.md Phase 5（更新プロトコル）を**省略禁止**として守る
- タスク1つ完了したら、即ファイル更新 → design-plan → design/README の順で同期
- 「あとでまとめて」は崩壊の第一歩

## 4. 横断的決定を個別ドキュメントだけに書く

**症状**: 「ID は ULID」が api-detail.md にだけ書かれている。data-layer.md を読んだ人は知らない。

**なぜダメか**: 設計書は **個別の深さ** と **全体の俯瞰** の両方が要る。横断決定が個別ドキュメントにしかないと、全体像が掴めない。

**回避**:
- 個別ドキュメントには**具体仕様**、design/README.md §重要な横断決定事項には**要約 + 出典**
- 両方に書くことで、深堀りもスキャンもできる

## 5. 実装寄りの詳細を書きすぎる

**症状**: 設計書に具体的な Python コードが30行連なる。

**なぜダメか**: 実装の細部は `python-impl` / `react-impl` の領分。設計書に入れると**実装が変わるたびに設計書も変える**ハメになり、形骸化する。

**回避**:
- 設計書には**型定義・関数シグネチャ・エラー時挙動**まで。実装本体は書かない
- 「このまま実装に落とせる」= コード骨格の粒度であって、完全コードではない

## 6. ユーザー承認なしに要求を追加する

**症状**: 「あとこれも必要ですね」と勝手に機能を足す（スコープクリープ）。

**なぜダメか**: スコープ管理はプロジェクト成否の根幹。勝手な追加は信頼を壊し、見積りを崩す。

**回避**:
- **必ず AskUserQuestion で確認** してから requirements.md に追加
- 「あとで議論する」と感じたら §11 未決事項 に置く
- 設計中に「これ要る」と気づいたら、一度手を止めて要求へ戻る

## 7. Mermaid 図で特殊文字を無警戒に使う

**症状**: シーケンス図のメッセージに `<br/>`、`"..."`、`;`、`{}` を混ぜたらパーサーエラー。

**なぜダメか**: Mermaid のパーサは特殊文字に弱い。エラーが出るとドキュメント全体が壊れて見える。

**回避**:
- `mermaid-tips.md` 参照。特殊文字は `Note over` に逃がす
- シーケンス図のメッセージは短く、記号は最小限

## 8. サーバー状態を Zustand に入れる設計を提案

**症状**: タスク一覧を TanStack Query で取得した後、Zustand ストアにコピーして使う。

**なぜダメか**: **キャッシュの二重管理**。Query が再取得しても Zustand は古いまま、あるいはその逆。同期コードを書き足すことになり、バグの温床。

**回避**:
- **サーバー状態は TanStack Query、UI 状態は Zustand** と決め切る（state-management.md §1）
- コンポーネントは直接 Query から読む。ストアは挟まない

## 9. 一度の AskUserQuestion で 5問以上聞く

**症状**: 10項目のチェックリストを一括で提示し、ユーザーがどこから答えればいいか迷う。

**なぜダメか**: 選択疲れで回答精度が落ちる。重要項目ほど雑に答えられる。

**回避**:
- **1ターン 2〜3問まで**。選択肢は 4つ以内
- ask-user-questions.md のステップ分割に従う
- 「決めきれない」なら即座に §11 未決事項 へ

## 10. 画面モックを本スキル内で作ろうとする

**症状**: ASCII アートや Mermaid で頑張って画面を描く。

**なぜダメか**: 視覚設計は視覚ツールに勝てない。中途半端な図は誤解を招く。

**回避**:
- D1.4 は **frontend-design プラグインに委譲**
- 別セッションで「画面モック作って」と指示して、本スキルは概念と設計の側に集中する
