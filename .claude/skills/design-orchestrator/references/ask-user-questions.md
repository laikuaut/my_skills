# 要求確定用の質問セット集

AskUserQuestion で使う質問の在庫。**鉄則**:

- **1セッション 2〜3問まで** まとめて問う（個別に連続ターンで聞くと会話が冗長化する）
- 選択肢は **4つ以内**。5つ以上は選択疲れを招く
- **推奨案を先頭**に置く。迷ったらこれ、という合意形成を加速させる
- 複数選択肢では「他（自由入力）」を末尾に置く
- 選んだ結果は**即座に requirements.md に反映**する（口頭合意で終わらせない）

## フェーズ別 質問カタログ

### Step 0: 輪郭（最初の1問）

- プロダクト名 / 1〜2行の説明 / 主ユーザー / 最優先する価値
- 最優先する価値の選択肢（multiSelect=false）:
  1. UI/UX品質（推奨）
  2. 機能数
  3. 性能
  4. プライバシー

### Step 1: 技術スタック（2問）

**Q1: フロントとバックの組み合わせ**
- 選択肢:
  1. React + FastAPI（推奨）
  2. Next.js + FastAPI
  3. React + Express
  4. 他（自由入力）

**Q2: データ保存**
- 選択肢:
  1. SQLite ファイル（推奨・小規模）
  2. PostgreSQL（本格運用）
  3. ローカルファイル（JSON / YAML）
  4. 他

### Step 2: 機能要件（2問）

**Q3: 認証の要否**
- 選択肢:
  1. 不要（ローカル利用のみ・推奨）
  2. パスワード + メール
  3. SSO（Google / GitHub）
  4. API Key のみ

**Q4: 対応言語（i18n）**
- 選択肢:
  1. 日本語のみ（推奨・v1）
  2. 日本語 + 英語
  3. 多言語（3つ以上）
  4. 後で決める

### Step 3: データ移行（2問）

**Q5: エクスポート・インポート**
- 選択肢:
  1. 両対応（JSON）（推奨）
  2. エクスポートのみ
  3. 不要
  4. CSV 対応が必須

**Q6: インポート衝突解決**（インポート対応時のみ）
- 選択肢:
  1. 新ID採番（安全・推奨）
  2. 既存を上書き
  3. ユーザーに選ばせる
  4. エラーで中断

### Step 4: 運用（2問）

**Q7: デプロイ方針**
- 選択肢:
  1. ローカル実行のみ（推奨・v1）
  2. 自前 VPS + Docker
  3. Vercel / Fly.io 等の PaaS
  4. 未定

**Q8: 想定ユーザー数**
- 選択肢:
  1. 1人（自分だけ・推奨）
  2. 数人（家族 / チーム）
  3. 数十〜数百
  4. 数千以上

### Step 5: 初期データ・細部（2問）

**Q9: 初期データ**
- 選択肢:
  1. 空（推奨）
  2. サンプル同梱（タグ4件 + タスク8件 等）
  3. デモモード切替可
  4. 未定

**Q10: タグ色プリセット**（タグ機能がある場合）
- 選択肢:
  1. 8色プリセット（推奨）
  2. カラーピッカーで自由選択
  3. 無色のみ
  4. 未定

### Step 6: 実装都合（1〜2問）

**Q11: バックエンド単独起動の可否**
- 選択肢:
  1. 可能（API テスト容易・推奨）
  2. 不可（フロント前提）
  3. 未定

**Q12: MCP 連携**
- 選択肢:
  1. 不要（推奨・v1）
  2. 必要（Claude Desktop 連携）
  3. 後で判断

## 質問セットの運用

1. **段階的に出す**。要求フェーズで一気に全部 = 選択疲れ
2. **選ばせすぎない**。決め切れない項目は「未決事項」（requirements.md §11）に残す
3. **結果を表で即可視化**。§10 確定事項を更新してからユーザ確認を取る
4. **根拠を書く**。特にデフォルトから外れた選択は、§10.1 に一言理由を残す
