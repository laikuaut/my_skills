# セッション分割の詳細ルール

設計フェーズを**複数セッションに分けて進める**ための判断基準。

## なぜ分けるのか

1. **コンテキスト肥大**: 13ドキュメント × 詳細仕様を1セッションで扱うと、後半になるほど粒度が落ちる
2. **頭の使い方の違い**: 視覚設計とシステム設計は別の思考モード。切り替えはコスト高
3. **途中保存による安定**: セッション終了時にコミットすると、問題発生時の戻し先が明確

## 区切り候補と判定

| タイミング | 判定 | 理由 |
|-----------|------|------|
| 要求仕様確定後 | 区切り候補 | 要求 → 設計の遷移は自然な節目 |
| D1.1 + D2.1 完了後 | 区切り候補 | 最初の2本は並行作業、完了で方向性確定 |
| システム設計（D2.x）完了後 | 区切り候補 | バック系の情報がまとまった状態 |
| API設計（D3.x）完了後 | **強い区切り推奨** | 設計書の半分が完了、実装に進める目処が立つ |
| D1.2〜D1.5 着手前 | **新セッション推奨** | 視覚設計は `frontend-design` と親和的 |
| D1.4 画面モック | **専用セッション** | `frontend-design` プラグインで別途実施 |
| D4.x MCP系 着手前 | 区切り候補 | MCP は独立した関心事 |
| D5 画面仕様書 | 区切り候補 | 実装後にリバース版で更新するため暫定で十分 |

## 定量的な基準

- **コンテキスト使用量が 40% を超えたら**切替を検討
- **15ターンを超えたら**一度仕切り直しを考える
- **ユーザの応答が短くなってきたら**疲労のサイン、区切りを提案

## セッション終了時のチェックリスト

区切る前に以下を完了させる:

- [ ] 今回のタスクの成果物が全てファイル化されている
- [ ] `design-plan.md` の進捗サマリが更新されている
- [ ] `design/README.md` の一覧と §横断決定事項 が更新されている
- [ ] 横断決定が発生していたら `design/README.md §重要な横断決定事項` に反映
- [ ] `git-commit-push` スキルで変更をコミット
- [ ] 次セッションで最初に着手するタスクIDを決めてある

## 新セッション開始時の始め方

新しいセッションで本スキルを再起動したら、以下を最初に読ませる:

1. `docs/requirements.md`（§10 確定事項 + §11 未決事項）
2. `docs/design-plan.md`（進捗サマリと「確定した重要事項」）
3. `docs/design/README.md`（§重要な横断決定事項）

この3本を最初に読めば、前セッションの成果を引き継げる。

## セッション間の引継ぎ情報

以下は必ずファイルに残す（記憶に頼らない）:

- **確定した要求**: requirements.md §10
- **未決の要求**: requirements.md §11
- **確定した設計決定**: design-plan.md の「Dx.y で確定した重要事項」
- **横断決定**: design/README.md §重要な横断決定事項
- **次にやること**: design-plan.md §次にやること

## 並行作業の扱い

- **D1.1 と D2.1 は並行可**だが、1セッション内で両方やるのは避ける（混乱する）
- **別セッションで同時進行**はOK（例: ユーザーが別タブで frontend-design を使う）
- 並行時は、**横断決定は完了した方が勝ち**（先に決めた側を正として、後発が追従）

## 分割の失敗パターン

- ❌ タスク途中で切る → 再開時に「どこまでやったっけ」でロス
- ❌ 引継ぎメモなしで切る → 次セッションが前提を復元できない
- ❌ コミットせずに切る → 物理的にロストするリスク

区切る = **成果物の保存 + 進捗更新 + コミット** の3点セットで完結させる。
