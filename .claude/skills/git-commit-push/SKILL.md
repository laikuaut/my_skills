# Git Commit & Push

差分チェックからpushまでを一気通貫で行うスキル。変更の意図を読み取り、適切な粒度でコミットを分割する。

## 全体フロー

1. 差分チェック
2. コミット分割の判断
3. コミットメッセージ生成
4. ユーザへの提示・確認
5. コミット実行
6. プッシュ確認・実行

---

## Step 1: 差分チェック

以下のコマンドで現在の状態を**漏れなく**把握する:

```bash
git status
git diff
git diff --staged
git log --oneline -5
```

全ての変更を3カテゴリに分類して一覧化する:
1. **ステージ済みの変更** — `git diff --staged` で確認
2. **未ステージの変更（追跡中ファイル）** — `git diff` で確認
3. **未追跡の新規ファイル** — `git status` の Untracked files で確認。各ファイルの中身を `cat` 等で読む

全ファイルを把握した後で、センシティブファイルのチェックを行う:
- `.env`、`credentials`、秘密鍵（`*.pem`, `*.key`）、`*secret*` などが含まれていたら警告し、コミット対象から**除外**する
- センシティブファイルを除外しても、残りの変更ファイルは全てコミット対象として扱う（除外したファイルにつられて他の変更を見落とさないこと）

## Step 2: コミット分割の判断

全ての変更内容（`git diff` および `git diff --staged`、未追跡ファイル）を読み、以下の基準でコミットを分割するか判断する:

**分割する場合:**
- 異なる機能・目的の変更が混在している（例: バグ修正と新機能追加）
- フロントエンドとバックエンドで独立した変更がある
- リファクタリングと機能変更が混在している
- 設定ファイルの変更とコードの変更が独立している

**分割しない場合:**
- 全ての変更が1つの機能・目的に関連している
- 変更ファイル数が少なく、全体で1つのまとまった変更である
- テストコードとその対象コードのペア

分割する場合は、関連するファイルをグループにまとめて、コミット順序を決める（基盤的な変更 → それに依存する変更の順）。

### 分割の具体例

| 変更内容 | 分割判断 | 分割例 |
|---------|---------|--------|
| `auth/login.py`（バグ修正）+ `users/utils.py`（ヘルパー追加） | 分割する | (1) fix: ログイン認証の不具合修正 / (2) add: ユーザヘルパー関数追加 |
| `routers/products.py` + `tests/test_products.py`（同じ機能のテスト追加） | 分割しない | feat: 商品エンドポイント追加（テスト含む） |
| `frontend/Login.tsx` + `backend/auth.py`（連携機能の追加） | 分割する | (1) add: 認証API追加（依存元）/ (2) add: ログイン画面追加（依存先） |
| `pyproject.toml`（バージョン更新）+ `app/main.py`（ロジック修正） | 分割する | (1) chore: 依存パッケージ更新 / (2) fix: メイン処理の不具合修正 |
| 同一ファイル内のリファクタリング + 新機能追加 | 分割する（可能なら） | (1) refactor: 〜の整理 / (2) feat: 〜機能追加 |

## Step 3: コミットメッセージ生成

各コミットに対して**2行**のメッセージを生成する:

**フォーマット:**
```
1行目: <接頭辞>: <変更の要約>（日本語、50文字以内）
2行目: <変更の詳細・理由>（日本語、72文字以内）
```

### 接頭辞ルール

1行目には必ず以下の接頭辞を付ける。変更内容に最も合うものを選ぶ:

| 接頭辞 | 用途 |
|---|---|
| fix | 既存機能のバグ修正 |
| hotfix | 緊急の修正 |
| add | 新しいファイルや機能の追加 |
| feat | 新機能の追加 |
| update | 既存機能の改修（問題はないが改善したい場合） |
| change | 仕様変更に伴う修正 |
| clean/refactor | コードの整理・改善 |
| improve | コードの改善 |
| disable | 機能の一時無効化 |
| remove/delete | ファイルや機能の削除 |
| rename | ファイル名の変更 |
| move | ファイルの移動 |
| upgrade | バージョンアップ |
| revert | 以前のコミットへの巻き戻し |
| docs | ドキュメントの修正 |
| style | コーディングスタイルの修正 |
| perf | パフォーマンス改善 |
| test | テストコードの追加・修正 |
| chore | ビルドツール・ライブラリの自動生成物、その他 |

### コミットメッセージ例

**例1:**
```
feat: ユーザ認証機能の追加
JWTトークンによるログイン・ログアウト処理を実装
```

**例2:**
```
perf: 商品一覧の描画速度を改善
仮想スクロールの導入により大量データ表示時のパフォーマンスを向上
```

**例3:**
```
docs: READMEのセットアップ手順を更新
Docker Compose対応に伴い環境構築手順を修正
```

**例4:**
```
fix: ログイン時のトークン検証エラーを修正
有効期限チェックのタイムゾーン不一致を解消
```

メッセージは変更の「何を」「なぜ」が伝わるように書く。diffの内容から意図を読み取り、単にファイル名を列挙するのではなく、変更の目的を表現する。

## Step 4: ユーザへの提示

コミット計画をテキストで提示した後、ユーザに承認を求める（AskUserQuestionで「実行する / メッセージ修正 / 分割を変更」などを選択させる）。

まずコミット計画を以下のフォーマットでテキスト出力する:

```
## コミット計画

### コミット 1/N
**メッセージ:**
> 1行目
> 2行目

**対象ファイル:**
- path/to/file1.ts (変更)
- path/to/file2.ts (新規)
- path/to/file3.ts (削除)

**変更概要:** 〜の機能を追加

---
（複数コミットの場合は繰り返し）

### Push先: origin/branch-name
```

次にユーザに確認する（AskUserQuestion使用例）:
- question: 「この内容でコミット・プッシュしてよいですか？」
- options: 「実行する」「メッセージを修正」「分割を変更」など適切な選択肢を提示

ユーザが修正を求めた場合（メッセージの変更、分割方法の変更など）は、それに従って調整する。

## Step 5: コミット実行

ユーザの承認を得たら、各コミットを順番に実行する:

```bash
# コミット1
git add <files>
git commit -m "$(cat <<'EOF'
1行目

2行目
EOF
)"

# コミット2（分割した場合）
git add <files>
git commit -m "$(cat <<'EOF'
1行目

2行目
EOF
)"
```

- `git add`はファイルを個別に指定する（`git add .`は使わない）
- コミットメッセージの1行目と2行目の間には空行を入れる（gitの慣例）
- pre-commitフックが失敗した場合は、原因を調査して修正し、新しいコミットを作成する（`--amend`は使わない）
- コミット失敗時は即座にユーザに報告する

## Step 6: プッシュ

コミット完了後、ユーザにプッシュの確認を行う（AskUserQuestion使用例）:
- question: 「コミットが完了しました。origin/branch-name にプッシュしますか？」
- options: 「プッシュする」「プッシュしない」など適切な選択肢を提示

ユーザが承認したら:

```bash
git push origin <branch-name>
```

- リモートブランチが存在しない場合は `git push -u origin <branch-name>` を使う
- main/masterへの直接プッシュの場合は特に注意を促す
- force pushは行わない
- プッシュ完了後、結果を報告する

## 注意事項

- コミット・プッシュの各ステップでユーザの確認を取ることが最重要。自動で進めない
- セキュリティに関わるファイル（.env、credentials、秘密鍵など）を検知した場合は必ず警告する
- `--no-verify`や`--force`は使用しない
- gitの設定（user.name, user.email等）は変更しない
