# handoff.md テンプレート

`docs/handoff.md` として配置する。セクション順序は固定（次セッションが機械的に読めるように）。

```markdown
# セッション引継ぎ
作成日時: {YYYY-MM-DD HH:MM}
元セッションの目的: {このセッション開始時のユーザ指示を 1-2 行で}
計画ソース: {docs/design-plan.md / docs/tasks.md / 会話履歴}

## 本セッションで完了したタスク
| ID | タイトル | 担当スキル | 成果物 |
|----|---------|----------|-------|
| T1 | ... | ... | ... |
| T2 | ... | ... | ... |

## 中断したタスク
### {ID}: {タイトル}
- **進捗**: {何がどこまで済んでいるかを具体的に}
- **残作業**: {次セッションで最初にやるべき具体的な手順}
- **参照ファイル**: {そのタスクで編集・参照していたファイル一覧}
- **注意点**: {ハマりポイントや判断保留事項があれば}

## 残タスク一覧（優先度順）
| ID | タイトル | 担当スキル | 優先度 | 依存 | 並列グループ |
|----|---------|----------|-------|-----|------------|
| T5 | ... | ... | must | T4 | — |
| T6 | ... | ... | must | T4 | parallel-group-A |
| T7 | ... | ... | should | — | parallel-group-A |

## 次セッション起動プロンプト（コピペ用）

\`\`\`
task-orchestrator を使って作業を継続してください。
前セッションの引継ぎは docs/handoff.md にあります。

最初にすべきこと:
1. docs/handoff.md を読み込む
2. 「{中断タスクID: タイトル}」から再開する
3. 残タスクの並列グループ {parallel-group-A など} を確認して進める

現在のブランチ: {ブランチ名}
元セッションのコミット: {最後のコミットハッシュ or "未コミット"}
\`\`\`

## 参照すべきファイル
- docs/requirements.md
- docs/design-plan.md
- {本セッションで読み書きした重要ファイル}

## 未解決の判断事項
- [ ] {ユーザ承認待ちの項目}
- [ ] {設計上まだ決まっていない点}

## 横断的な決定事項（本セッション分）
- {設計原則・アーキテクチャ方針・命名規約など、複数タスクに影響する決定}
- {例: "フォーム状態は React Hook Form を採用"}
- {例: "API エラー時はトースト表示で統一"}

## 既知の懸念点
- {今後問題になりそうな点、注意すべき点}
- {未調査だが気になる領域}
```

## 記入時の注意

- **再開手順は「次セッションがファイルだけ見て再開できる」粒度**で書く。「サービス層を実装する」ではなく「`backend/src/services/user_service.py` を新規作成し、Pydantic `UserCreate` を受け取り DB にINSERT するメソッドを追加する」レベルまで具体化する。
- **横断的な決定事項は必ず記載**。個別タスクの決定は各タスクの成果物内で管理されるが、プロジェクト全体に効く決定はここに集約しないと次セッションが再発明する。
- **参照すべきファイルは絶対に網羅的に**書く。リンク切れや見落としがあると次セッションがまた探索から始まる。
- **コピペ用プロンプトは自己完結**に。handoff.md を読む前提でも、「何を最初にすべきか」を数ステップで書いておく。

## 最小テンプレート（小規模セッション用）

フェーズ 1 つだけのような小さなセッションでは以下で十分:

```markdown
# セッション引継ぎ ({YYYY-MM-DD HH:MM})
完了: {タスクID の羅列}
中断: {ID と再開手順 1 行}
次: {次セッション起動プロンプト のみ}
```

ただし **次セッション起動プロンプト は常に省略不可**。これが無いと handoff.md の存在意義が無い。
