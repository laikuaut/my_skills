# 計画ソースの検出・構造化・統合

## 検出の優先順位

1. `docs/design-plan.md` — `design-orchestrator` 生成の公式計画
2. `docs/tasks.md` / `TODO.md` / `tasks.md` — 自由形式
3. 直近の会話履歴（Plan モード出力 / ユーザが口頭で並べたタスク列）
4. いずれも無い → AskUserQuestion で策定して `docs/tasks.md` に書き出す

## 各ソースの読み方

### design-plan.md

`design-orchestrator` が生成するフォーマットを前提にする。典型的な構造:

```markdown
## 進捗サマリ
| ID | タスク | 状態 | 担当 | 備考 |
|----|-------|------|------|------|
| D1.1 | UI/UXコンセプト | ✅ | - | 完了 |
| D1.2 | デザイントークン | ⏳ | - | 作業中 |
| D2.1 | アーキテクチャ | ☐ | - | 未着手 |
```

- `✅` / `完了` / `[x]` → status=done
- `⏳` / `作業中` / `[~]` → status=in_progress
- `☐` / `未着手` / `[ ]` → status=pending
- `🚫` / `ブロック` → status=blocked

ID は `D1.1` `D2.3` など階層型。依存関係は `design-orchestrator` の「タスク推奨順序」から推定する（同スキルの SKILL.md 参照）。

### tasks.md / TODO.md

自由形式なので典型的なパターンを拾う:

```markdown
- [ ] バックエンドのユーザーAPIを実装 [parallel-group-A]
- [ ] フロントエンドのユーザー一覧画面 [parallel-group-A]
- [x] DBマイグレーション
- [ ] E2Eテスト（depends: バックエンド, フロント）
```

- `[ ]` → pending, `[x]` → done
- `[parallel-group-X]` → parallel_group=X
- `(depends: ...)` → depends_on を推定
- 順序は上から順に priority=高

### 会話履歴

ユーザが「まずAやって、次にB、そのあとCとDは並列でいい」と口頭で言っている場合など。**必ず構造化してユーザに確認を取る** こと。推定だけで進めると解釈違いが起きる。

構造化した後、ユーザが同意したら `docs/tasks.md` に書き出して以降はそれを正とする（会話履歴は揮発性が高いため）。

## 複数ソースが併存する場合

最頻出のパターン: `design-plan.md` と `tasks.md` が両方ある。

判断基準:

- 設計フェーズの途中なら → `design-plan.md` を正とする
- 設計完了後の実装フェーズなら → `tasks.md` を正とする（or 新規作成）
- 両方に同じタスクがある場合 → どちらを正とするかユーザに確認

**必ず 1 ソースに絞る**。二重管理は進捗破綻の元。

## 計画が見つからない場合

AskUserQuestion で以下を確定する:

1. 何を作るか / やるか（1 行）
2. 想定作業時間（今セッションで完了したいか / 複数セッションか）
3. 優先順位

回答を踏まえて `docs/tasks.md` を作成し、ユーザ承認のうえ以降はそれを使う。

## タスク ID の扱い

- 既存 ID があればそれを使う（`D1.1` `T5` など）
- 無ければ `T1` から付番する
- 計画ファイル側にも同じ ID を書き戻す（後続セッションが参照できるように）
