# セッション限界判定の詳細ルール

## 基本思想

Claude のセッションは **コンテキスト肥大** と **注意力低下** で徐々に品質が落ちる。早すぎる切替は手間が増えるだけ、遅すぎる切替は引継ぎ情報の欠落を招く。この中間点を狙う。

**ハイブリッド判定** の名の通り、機械的ルールと Claude 自身の内省を組み合わせる。

## ルールベース（必ずチェック）

以下のいずれかに該当したら「区切り候補」フラグを立てる:

### R1. フェーズ境界

- 論理設計フェーズ（`design-orchestrator`）完了
- API 設計フェーズ完了
- バックエンド実装完了
- フロントエンド実装完了
- E2E テスト完了
- 1 機能フルスタック実装（`fullstack-pipeline` 一巡）完了

### R2. タスク完了数

- 5 タスク以上連続で完了

### R3. ユーザの明示シグナル

- 「重い」「遅い」「切り替えたい」「一旦切ろう」「次セッションで」

### R4. 行き詰まりシグナル

- 同じエラーで 3 回以上リトライ
- ユーザが同じ指摘を繰り返す
- 想定外の依存関係が連続発覚

### R5. ドメイン切替シグナル

- デザイン系 ⇔ システム系 の切替が次タスクで発生
- 全く異なる機能領域への切替（例: 認証機能 → 分析ダッシュボード）
- 使用する子スキルが大きく変わる（例: python-impl → frontend-design）

## 自己申告（内省判定）

区切り候補フラグが立った時点で以下を自問:

### S1. コンテキスト肥大感

- 大量のファイルを読んだ感覚があるか？
- 長い tool 結果（特にログ、検索結果）が複数含まれているか？
- 計画ファイル以外の「参考資料」が溜まっていないか？

### S2. 次タスクの切替コスト

- 次タスクが新しいドメインに入るか？
- 新しい子スキルを発火させる必要があるか？
- まだ読んでいない設計書を参照する必要があるか？

### S3. 残容量の感覚

- あと 1 タスク走らせる余裕があるか？
- もしユーザから修正依頼が来ても対応できる余裕があるか？

## 判定マトリクス

| ルール該当 | 自己申告 | 判定 |
|---|---|---|
| R1/R5 該当 | どちらでも | **切替推奨**（次セッションへ） |
| R2 該当 | S1/S2 で Yes | **切替推奨** |
| R2 該当 | S1/S2 で No | 続行（あと 2-3 タスクで再評価） |
| R3 該当 | どちらでも | **即切替**（ユーザ意思優先） |
| R4 該当 | どちらでも | **切替推奨**（環境リセットで解消することも多い） |
| 該当なし | S1 で Yes | 切替検討（ユーザに軽く提案） |

## 例外

### 切替しないほうが良い場合

- **進行中タスクの途中** — 1 タスクを完走してから切る
- **ユーザ承認待ち** — 回答をもらってから切る
- **テスト実行中 / ビルド中** — 結果を見てから判断
- **コミット直前** — コミットしてから切る（handoff.md も一緒にコミット）

### 切替を強く推奨する場合

- **画面モック作成タスクの前** — `frontend-design` プラグインは独立セッションが基本
- **大規模リファクタの前** — コンテキストをクリアにして臨む方が精度が高い
- **ユーザが明らかに疲れているシグナル** — 「もうやって」「適当で」などの投げやりな発言

## 切替時のユーザへの伝え方

区切り推奨に達したら以下のような声がけをする:

```
ここまで {N} タスク完了しました。次の {タスク名} はドメインが切り替わる
（{切替内容}）ので、セッションを一旦区切ることをおすすめします。

引継ぎメモ（docs/handoff.md）を作成し、新セッションで再開できる形に
しますがよろしいですか？
（続行する場合は "続行" と返信）
```

ユーザが続行を選んだ場合は、次の区切り候補まで進む。引継ぎ生成を強制しない。
