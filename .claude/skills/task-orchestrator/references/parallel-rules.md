# 並列実行判定ルール

## 基本方針

**明示タグ最優先 → 推定 → ユーザ承認** の順で判定する。

並列起動は生産性向上に寄与するが、同じファイルを触ると容易に競合する。**疑わしい時は逐次** が基本姿勢。

## 明示タグ記法

計画ファイル側で以下のいずれかを書いてあれば尊重する:

```markdown
# パターン A: 行末タグ
- [ ] T5: バックエンドAPI実装 [parallel-group-A]
- [ ] T6: フロント Types 定義 [parallel-group-A]

# パターン B: 見出し直下の属性記法
## T5 バックエンドAPI実装
parallel_group: A

# パターン C: YAML フロントマター付き計画
tasks:
  - id: T5
    parallel_group: A
```

同じグループ名のタスクは同時に起動可能、異なるグループは逐次。

## 推定ルール（タグが無い場合）

以下をすべて満たせば並列候補:

| ルール | 例 |
|---|---|
| 互いに依存関係に無い | T5 が T4 の結果を待つ → T5 と T4 は NG |
| 触るディレクトリが重ならない | `backend/` と `frontend/` は OK |
| 同じファイルを書き換えない | 両方とも `docs/README.md` を更新 → NG |
| 出力が他方の入力にならない | スキーマ定義（T3）→ バックエンド実装（T5）は逐次 |
| 同じ子スキルを動かさない（推奨） | 同じスキルを並列起動するとプロジェクト状態が重なりやすい |

## 並列起動の典型パターン

### OK の例

- `backend/` 実装 × `frontend/` 実装（インターフェースが確定済みの場合）
- バックエンドテスト作成 × フロントエンドテスト作成
- ドキュメント生成 × テスト実装（どちらも読み取り中心）
- 独立した複数機能の同時着手（例: ユーザー管理 × 通知機能 × 検索）

### NG の例

- API スキーマ定義 × バックエンド実装 → スキーマが先
- バックエンド実装 × OpenAPI 生成 → 実装が先
- 実装 × E2E テスト → 実装が先
- 両方が `package.json` や `pyproject.toml` を触る → 衝突しやすい

## ユーザ承認の取り方

必ず並列起動**前**に以下のような確認を入れる:

```
次は T5「バックエンドAPI実装」と T6「フロント Types 定義」を並列起動します。
- T5: backend/src/ を編集
- T6: frontend/src/types/ を編集
触るディレクトリが重ならないので並列可と判断しました。

この並列化で進めていいですか？（逐次にする場合は "逐次で" と返信）
```

## 並列起動の具体的な呼び出し方

メインセッションで 1 メッセージ内に複数の Agent ツール呼び出しを並べる。ただし **個別実装タスクはメインセッションで子スキルを発火** させた方が文脈が保てるので、サブエージェント起動は探索系に限る。

実装タスクを並列化したい場合:

- 方法 1: **タスクを順に 2 つ走らせる**（見かけ上の並列化は諦め、1 つずつメインで対応）
- 方法 2: **片方をサブエージェントに投げる**（例: "フロントの types 定義はサブで並行して" と切り出し、メインはバックエンドに集中）

方法 2 を選ぶ場合、サブエージェントには **タスク内容と担当スキル名（発火させるトリガー表現）** を明示して渡す。

## 並列度の上限

- 同時並列は **最大 3 タスク** まで（それ以上はコンテキスト管理が破綻しやすい）
- 並列実行中は新規タスクの着手を待つ（途中で割り込ませない）

## 失敗時の扱い

並列起動した片方が失敗した場合:

1. 成功した方の成果物は残す
2. 失敗した方はロールバック or 手動修復
3. 次のループから並列化の基準を厳しくする（同じパターンで再失敗しないように）
