# タスクオーケストレータ

計画に沿って、**子スキルを選んで起動する**最上位メタ制御スキル。
実装・設計・テスト・ドキュメント生成などの具体作業は一切行わず、次の 5 点だけをひたすら管理する。

1. 次に何をやるか
2. 誰（どのスキル）に任せるか
3. 並列でいけるか、逐次が安全か
4. いつ区切るか（セッション限界判定）
5. どう引き継ぐか（handoff.md + チャット要約）

`design-orchestrator` が "論理設計フェーズ" を、`fullstack-pipeline` が "実装フェーズ" を指揮するのに対し、本スキルは **その両者の上位** に立ち、**複数フェーズ・複数機能にまたがる全体工程** を指揮する。

## 責務分担

| このスキル | 委譲先 |
|---|---|
| 計画ソースの検出・構造化 | — |
| 並列実行可能性の判定 | — |
| 次タスクの選択と委譲指示 | — |
| セッション限界判定・引継ぎ生成 | — |
| 計画ファイルの進捗更新 | — |
| 論理設計ドキュメントの作成 | design-orchestrator |
| フルスタック1機能の実装指揮 | fullstack-pipeline |
| Python / React の実装 | python-impl / react-impl |
| プロジェクト雛形生成 | python-project-init / react-project-init |
| OpenAPI / APIテスト / 画面仕様書 / E2E | openapi-docs / rest-api-test / screen-spec-reverse / playwright-test |
| コミット・プッシュ | git-commit-push |

**鉄則**: 本スキルは Edit / Write をタスク実行には使わない（**handoff.md と計画ファイルの進捗更新のみ** 例外的に使用）。コード変更・設計書作成は必ず子スキルに委譲する。自分でやってしまうと子スキルの品質基準（docstring / 型ヒント / コンポーネント分割など）が効かなくなる。

---

## 全体フロー

```
Step 1: 計画ソース検出
         ↓
Step 2: タスク構造化（ID / 依存 / 並列タグ / 担当スキル推定）
         ↓
Step 3: ユーザへ提示・承認取得
         ↓
Step 4: 実行ループ
  ├─ 次タスク選択（依存解消済み・最高優先）
  ├─ 並列可能なら ユーザ承認のうえ 複数タスクを並列起動
  ├─ 担当子スキルに委譲
  ├─ 完了反映（計画ファイルを ✅ に）
  └─ セッション限界チェック
         ↓ (限界到達)
Step 5: 引継ぎ生成（handoff.md + チャット要約）
         ↓
Step 6: コミット（git-commit-push）
```

各タスク完了ごとに「次どうする？」と立ち止まらず、ユーザが明示的に止めるまでループを回すのが基本姿勢。ただしユーザ承認ポイント（並列起動前 / フェーズ境界 / セッション切替時）は必ず確認を挟む。

## Step 1: 計画ソース検出

スキル発火直後に **黙って検出だけ実行** し、最初のメッセージで結果を一括提示する（何度も往復しない）。

優先順位:

1. `docs/design-plan.md` — `design-orchestrator` が作った公式タスクリスト
2. `docs/tasks.md` / `TODO.md` / `tasks.md`（プロジェクトルートまたは docs/） — 自由形式
3. 直近の会話履歴 — ユーザが口頭で述べたタスク列や Plan モードの出力
4. 上記がいずれも無い — AskUserQuestion で一緒に計画を策定（この場合は `docs/tasks.md` として書き出す）

複数ソースが併存する場合は「どれを正とするか」を確認する。**後続の進捗更新先を 1 つに絞らないと二重管理で破綻する** ため。

詳細は `references/plan-sources.md`。

## Step 2: タスク構造化

検出した計画から以下を抽出する。長い計画でも **TodoWrite には落とさず頭の中で整理**（TodoWrite は本スキル自体の作業進捗用）。

| 属性 | 意味 |
|---|---|
| id | T1, T2, … （計画に番号があればそれを使う） |
| title | 短い表題 |
| source | 計画ファイル内の位置（行番号または章番号） |
| status | pending / in_progress / done / blocked |
| depends_on | 先行タスクの id 配列 |
| parallel_group | `[parallel-group-A]` 等のタグ（明示）または推定結果 |
| skill | 担当子スキル名（推定） |
| priority | must / should / nice |

**担当スキル推定** の主要パターンは `references/task-dispatch.md` にマッピング表がある。迷ったら同ファイルを参照。

## Step 3: ユーザへ提示

検出結果・構造化結果・次アクション案を **1 ターンで** 提示する。往復を減らすため、推奨順序と並列候補を先に全部出す。

**提示テンプレ**:

```
計画ソースを検出しました:
- {検出したファイルとパス}
- 総タスク数: {N}（うち未完了: {M}）

未完了タスクの推奨実行順:
1. T4「APIスキーマ定義」→ design-orchestrator
2. T5「バックエンドAPI実装」+ T6「フロントTypes定義」→ 並列可能（parallel-group-A: backend/ と frontend/ が独立）
3. T7「E2Eテスト」→ playwright-test

最初に T4 を始めます。その後 T5/T6 を並列起動したいですがよろしいですか？
あるいは別のタスクから始めますか？
```

ユーザが OK / 別順序 / 特定タスクだけ実行 などを指示した後に Step 4 へ。

## Step 4: 実行ループ

### 4-1: 次タスクを選ぶ

- 依存が解消済みの pending から、priority と計画上の順序で先頭を選ぶ
- 並列可能なタスクがあればグループで選ぶ

### 4-2: 並列実行可能性の判定

**ポリシー**: 明示タグ最優先、無ければ推定。**いずれの場合も並列起動前にユーザへ候補を提示して承認を取る**（ファイル競合リスクのため）。

#### 明示タグ

計画ファイルの行末または見出しに以下のような記法がある場合:

```markdown
- [ ] T5: バックエンドAPI実装 [parallel-group-A]
- [ ] T6: フロントエンド Types 定義 [parallel-group-A]
- [ ] T7: DBマイグレーション [parallel-group-B]
```

同じグループタグのタスクを同時に起動できると判断する。

#### 推定（タグが無い場合）

以下すべてを満たせば並列候補:

1. 互いに `depends_on` に入っていない
2. 触るディレクトリが重ならない（例: `backend/` と `frontend/` は独立）
3. 同じファイルを書き換えない
4. 一方の出力が他方の入力になっていない（スキーマ→実装などは NG）

**保守的に判断する**。疑わしい場合は逐次を選んだ方が事故が少ない。詳細は `references/parallel-rules.md`。

### 4-3: 委譲方法

- タスク内容から **「○○スキルに従って □□ を実装する」旨の明確な指示** をメインセッション上で出す
- メインセッションでそのスキルを **発火させる**（子スキルの description に合うトリガー表現を使う）
- サブエージェント（Agent ツール）起動は **独立探索が必要な場合のみ**（複数リポジトリ横断検索など）。通常の実装はメインセッションで子スキルを動かすほうが文脈が保てる

**並列起動する場合**: 同一メッセージ内で **複数の Agent ツール呼び出し** を並べる（Agent の subagent_type を適切に選ぶ）。逐次なら 1 件ずつメインで発火。

### 4-4: 完了反映

1 タスク完了したら **必ず** 以下を実行する（省略すると次セッションで未完了扱いに逆戻りする）:

1. 計画ファイルの該当行を `[ ]` → `[x]` または `✅` に
2. 重要な決定事項（横断的なもの）は計画ファイルの末尾に「本セッションの決定事項」として追記
3. チャットに 1 行で完了サマリ「[完了] T4 APIスキーマ定義」

### 4-5: セッション限界チェック

Step 5 で詳述。ループを 1 周するたびに判定する。

---

## Step 5: セッション限界判定（ハイブリッド）

### ルールベース（必ずチェック、いずれかに該当したら区切り候補）

- フェーズ境界を越えた（設計完了 / API 完了 / バックエンド完了 / フロント完了 / E2E 完了）
- 5 タスク以上連続で完了した
- ユーザが「重い」「遅い」「切り替えたい」を明示した
- 同じ箇所で 3 回以上リトライが発生した（行き詰まりシグナル）
- デザイン系 ⇔ システム系の切替が次タスクで発生する（視覚設計と論理設計は別セッションが効率的）

### 自己申告（内省判定）

区切り候補に達した or 達していなくても、以下を自問する:

- コンテキストが肥大している感覚があるか？（大量ファイルを読んだ / 長い tool 結果が多い）
- 次のタスクが新しいドメイン/スキルに切り替わるか？（切替時は新セッションの方が効率的）
- もう一つ大きなタスクを走らせられる余裕があるか？

**区切り候補 + 自己申告で Yes → Step 6（引継ぎ）へ**。
**区切り候補 + 自己申告で No → 続行**。
**区切り候補に該当せず → 続行**（ただし "次が新ドメイン" は強めのシグナルなので単独でも切ってよい）。

判定ルールの根拠と例外は `references/session-limits.md`。

## Step 6: 引継ぎ生成（ファイル + チャット要約）

**どちらか片方ではダメ**。ファイルは一次情報、チャット要約は次セッション起動のショートカット。

### 6-1: `docs/handoff.md` を作成

テンプレートは `references/handoff-template.md`。必須セクション:

- セッション要約（開始時の目的、実際にやったこと）
- 完了したタスク一覧（id / title / 成果物）
- 中断したタスク（現在地と再開手順を具体的に）
- 残タスク一覧（優先度・担当スキル付き）
- 未解決の判断事項（ユーザ承認待ち）
- 横断的な決定事項（設計上・実装上の重要メモ）
- 参照すべきファイル
- **次セッション起動プロンプト（コピペ用コードブロック）**

### 6-2: チャットに要約を表示

```
[引継ぎ] docs/handoff.md に保存しました。

新セッションで以下をコピペすれば継続できます:

== ここから ==
task-orchestrator を使って作業を継続してください。
前セッションの引継ぎは docs/handoff.md にあります。
最初に handoff.md を読み込み、「T6 バックエンドAPI実装（サービス層未着手）」から再開してください。
== ここまで ==

本セッションの要点:
- 完了: {N} 件（T1〜T5）
- 中断: T6（ルーター層まで完了、サービス層未着手）
- 未解決: なし / {項目}

次のステップ: この変更を git-commit-push でコミットすることをおすすめします。
```

チャット要約は **起動プロンプトを含む** ことが重要。ユーザがそのまま新セッションへコピペできる完成形にする。

---

## Step 7: コミット

区切りごとに `git-commit-push` を呼ぶ。分割の目安:

| 内容 | コミット単位 |
|---|---|
| 設計系ドキュメントの更新 | 1 コミット |
| バックエンド実装 | 1 コミット |
| フロントエンド実装 | 1 コミット |
| テスト追加 | 1 コミット |
| handoff.md + 計画ファイル進捗更新 | 最後に 1 コミット |

handoff.md は必ずコミットする（次セッションが読むため）。コミット漏れすると引継ぎが機能しない。

## 発火時の最初のメッセージ

スキル発火直後の応答テンプレ:

1. `Glob` で `docs/design-plan.md`、`docs/tasks.md`、`TODO.md`、`tasks.md` を一括検出
2. 該当ファイルを `Read`
3. 以下の形式で 1 ターン応答:

```
計画ソースを検出しました:
- {パス}: 未完了 {M} タスク
- {他のパス}: ...

推奨実行順:
  1. {T番号}「{title}」→ {skill}
  2. {並列候補があればここ}
  3. ...

最初に {先頭タスク} を始めていいですか？
あるいは別のタスクから / 並列候補を変更したい場合は教えてください。
```

計画が見つからない場合は AskUserQuestion で「何を作りたい？どんな順番？」を 2〜3 問で確定させて `docs/tasks.md` に書き出してから開始。

## アンチパターン

| やらない | 理由 |
|---|---|
| 計画ファイルを見ずに勝手にタスクを並べる | ユーザの計画と乖離して混乱 |
| 並列起動をユーザ承認なしで実行 | ファイル競合・意図せぬ変更のリスク |
| 自前で Edit / Write してタスクをこなす | 子スキルの品質基準（docstring / 型 / 分割）が効かない |
| セッション限界判定をスキップして走り続ける | 引継ぎ情報欠落で次セッションが詰む |
| handoff.md を書かずチャット要約だけで済ませる | 次セッションが読む一次情報を失う |
| チャット要約に起動プロンプトを含めない | 新セッション起動の手間が増える |
| コンテキストが軽いうちから引継ぎを乱発 | 生産性低下・中断疲れ |
| タスク完了時に計画ファイルを更新し忘れる | 次回起動時にまた未完了扱い |
| サブエージェントを乱発 | 会話の文脈分断・コスト増 |
| 計画ソースが複数あるのに統合せず両方更新 | 二重管理で進捗が破綻 |

## 既存スキルとの関係

| 既存スキル | 本スキルとの関係 |
|---|---|
| `design-orchestrator` | **子**（設計フェーズのタスクが並んでいたら本スキルから起動） |
| `fullstack-pipeline` | **子**（実装フェーズのタスクが並んでいたら本スキルから起動） |
| `python-impl` / `react-impl` | 子（個別実装タスク向け） |
| `python-project-init` / `react-project-init` | 子（プロジェクト雛形が必要なら起動） |
| `openapi-docs` / `rest-api-test` / `screen-spec-reverse` / `playwright-test` | 子（各専門領域） |
| `git-commit-push` | 子（区切りごとに呼ぶ） |

**関係図**:

```
[task-orchestrator] ← 最上位
   ├── [design-orchestrator]   ← 設計フェーズの指揮
   ├── [fullstack-pipeline]    ← 1機能フルスタック実装の指揮
   ├── [python-impl] / [react-impl] / ...  ← 個別実装
   └── [git-commit-push]        ← 区切りコミット
```

本スキルは `design-orchestrator` や `fullstack-pipeline` とレイヤーが違う。同じ粒度で競合しない。
